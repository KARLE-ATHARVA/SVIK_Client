# Visualizer Filter Flow Documentation

## Scope
- This document describes the active filter pipeline for legacy visualizer pages loaded as `/app/<roomId>.html` inside the Next.js route iframe.
- It reflects the current implementation where menu panel partials are UI shells only (no hardcoded tile `<li>` data).

## 1) Room Selection -> Visualizer Entry
- Room picker UI: `public/app/modules/partials/room.html`
- Room links open static visualizer pages like `6.html`, `20.html`.
- Next.js wrapper route: `src/app/[roomId]/page.tsx` (iframe host only).

## 2) Where Filter UI Comes From
- Filter modal skeleton is generated by:
  - `getCommonFilterModalsHtml()` in `public/app/modules/shared/modals.js`
- Sidebar panel shells are loaded from:
  - `public/app/modules/partials/menu-panel-1.html`
  - `public/app/modules/partials/menu-panel-2.html`
- Important (current state):
  - These panel files now contain only shared UI shell (buttons/search/empty `ul.tiles-list`).
  - They do **not** contain hardcoded tile list items anymore.

## 3) Runtime Data + UI Injection
- Main bootstrap module: `public/app/modules/data/dynamic-tiles-api.js`
- Called from room pages after both panel partials load:
  - `window.initDynamicTilesFromApi({ roomId })`
- This module:
  - Resolves `roomId -> spaceName` (living/kitchen/bathroom/etc).
  - Calls backend endpoints:
    - `FilterOptions?spaceName=...`
    - `FilterTileList?spaceName=...&appNames=...&colorNames=...`
  - Maps API rows to legacy tile schema (`avail_tiles`, `filters`, `size`, etc).
  - Builds tile `<li>` list HTML at runtime and injects into `#tiles-list-1` and `#tiles-list-2`.
  - Rebuilds filter option groups (size/finish/category/application/color).
  - Removes legacy groups (price/sanita).

## 4) Filter Execution Logic
- Core filtering (show/hide tile items) is in:
  - `public/app/modules/shared/filter-logic.js`
- `applyFilter(panelKey)` behavior:
  - Reads selected filter options from modal inputs (`data-filter-id`).
  - Matches each tile item using its `data-filter-*` attributes.
  - Shows item only if all active groups match.
- Rules enforced:
  - Application (`data-filter-id="34"`) is single-select.
  - At least one value remains selected per visible group.
  - Safe default is enforced when group becomes empty.

## 5) Legacy Compatibility Layer
- Legacy dropdown filter helpers remain in:
  - `public/app/visualizer/js/script-2.js` (`setTileFilter`, `fillCatSelB`, `fillCatSelSize`)
- Current runtime experience is driven primarily by:
  - dynamic option generation in `dynamic-tiles-api.js`
  - checkbox-based filtering in `filter-logic.js`

## 6) Backend/API Responsibilities
- External backend base URL:
  - `.env`: `NEXT_PUBLIC_API_BASE`
- APIs consumed by visualizer frontend:
  - `FilterOptions`
  - `FilterTileList`
  - `FilterAvailableOptions` (used in React stack, not legacy static room pages)
  - Favorites/Auth APIs (`list_fav`, `add_fav`, `remove_fav`, `cust_login`, `cust_signup`)
- In-repo API route:
  - `src/app/api/tile-image/route.ts` (image proxy only; not filter business logic).

## 7) Current Simplification Outcome
- Removed duplication and stale hardcoded catalog data from both menu panel partials.
- Source of truth for tile/filter data is now runtime API response + mapping logic.
- UI shell is shared in structure across panel 1 and panel 2, with only panel-specific IDs/targets.
